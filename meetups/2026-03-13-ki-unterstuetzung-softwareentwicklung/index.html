Bereich 1

Die Evolution der Entwicklungsunterstützung

Vom blinkenden Cursor ohne jede Hilfe bis zur KI, die eigenständig plant, codet und testet – so hat sich unsere Entwicklungsunterstützung verändert.

1Einfacher Texteditor
vi – users.txt
~
~
function getUser(id) {
  var result = db.query("SELECT * FROM users WHERE id=" + id);
  return result;
}
~
~
-- INSERT -- line 3, col 1

Kein Syntax-Highlighting, keine Hilfe. Jedes Zeichen muss man selbst tippen. Wer einen Fehler macht, merkt es erst beim Kompilieren.

2IDE mit Hilfsfunktionen
IDE – userService.js
function getUser(id) {
  var result = db.query(
    "SELECT * FROM users WHERE id=" + id
  );
  return result;
}

Syntax-Highlighting! Mehrere Zeilen gleichzeitig editieren, Code-Vergleiche, integrierter Compiler. Fehler werden sofort angezeigt.

3Starke IDE mit Sprachverständnis
WebStorm – userService.js
function getUser(id) {
  return userRepository.findBy
    ▶ findById(id): User
    findByName(name): User[]
    findAll(): User[]
    count(): number
}

Autovervollständigung kennt alle Methoden und Typen. Verlinkung zu Funktionsdefinitionen, Refactoring über alle Dateien, Fehler vor dem Kompilieren.

4Rudimentäre KI: Zeilenvorschläge
VS Code + TabNine – userService.ts
function getUser(id: number): User {
  return userRepository.findById(id);
}

function deleteUser(id: number): void {
  userRepository.delete(id);
}

Die KI schlägt den Rest der Zeile vor – basierend auf statistischer Wahrscheinlichkeit. Fühlt sich an wie Magie, ist aber noch limitiert.

5Quellcode-Verständnis
VS Code + Copilot – userService.ts
// Finde User mit Validierung und Error-Handling
async function getUser(id: string): Promise<User> {
  if (!id) throw new BadRequestError('ID required');
  const user = await userRepository.findById(id);
  if (!user) throw new NotFoundError(`User ${id} not found`);
  return user;
}

Ein Kommentar reicht – die KI generiert ganze Funktionen mit Validierung, Error-Handling und passenden Typen. Sie versteht den Kontext der Datei.

Cursor – 3 Dateien gleichzeitig
user.model.ts

  name: string;
  email: string;
}
user.service.ts

  async create(dto: CreateUserDto) {
    return this.repo.save(dto);
  }
}
user.test.ts
describe('UserService', () => {
  it('should create user', ...
  it('should validate email', ...

Die KI versteht das Projekt. Eine Anweisung – und Model, Service und Tests werden zusammenhängend generiert. Mehrere Dateien, ein Plan.

Aufgabe verstehen → coden → testen → debuggen → optimieren. Die KI arbeitet in einer Schleife – wie ein Entwickler. Aber: kleines Kontextfenster = senil nach langen Sessions.

8Senior-KI-Entwickler
Claude Code 2.0 – Agent Teams
$ claude "Implementiere das komplette Ticket LT-428"

▶ Memory geladen: Projektstruktur, Konventionen, letzte Session
▶ Ticket analysiert: User-Rollen mit RBAC
▶ Agent 1: Backend – Model, Service, Guards
▶ Agent 2: Frontend – Pages, Composables, Forms
▶ Agent 3: Tests – E2E + API-Tests
▶ Review-Agent prüft Code-Qualität... ✓
▶ 12 Dateien, 28 Tests, alle grün.
▶ Session: 45 Min. | Memory gespeichert.

Großes Kontextfenster, Memory über Sessions hinweg, mehrere Agents parallel. Plant wie ein Senior, kennt das Projekt, arbeitet stundenlang – und merkt sich alles.

METR-Studie: Die Wahrnehmungslücke

In einer kontrollierten Studie brauchten Entwickler mit KI-Tools tatsächlich 19% länger bei komplexen Real-World-Tasks – glaubten aber, 20% schneller gewesen zu sein. Ein Follow-up (Feb. 2026) bestätigt: Die Wahrnehmungslücke bleibt – neuere Tools verbessern sich, aber der Effekt ist schwer messbar.

Begriffsklärung

Vom Vibe-Coder zum Agentic Engineer

Vibe-Coding klingt cool – aber es beschreibt nur die halbe Wahrheit. Was wir tatsächlich tun, ist etwas anderes.

🎸
Vibe-Coder
  • ❌ Prompt rein, Code raus – fertig
  • ❌ Kein Review, keine Tests
  • ❌ „Funktioniert irgendwie“
  • ❌ Versteht den generierten Code nicht
  • ❌ Keine Qualitätskontrolle
  • ❌ Hofft, dass es hält
🧑‍💻
Agentic Engineer
  • ✅ Klare Aufgabenstellung & Akzeptanzkriterien
  • ✅ TDD – Tests vor dem Code
  • ✅ KI als autonomer Agent mit Leitplanken
  • ✅ Versteht & reviewt jeden Output
  • ✅ Regeln, Prozesse & Plugin als Wissensbasis
  • ✅ Human in the Loop für Entscheidungen

„Agentic Engineering bedeutet: Die KI arbeitet autonom – aber nach deinen Regeln, mit deinen Tests und unter deiner Aufsicht. Das ist der Unterschied zwischen Chaos und Produktivität.“

Die zentrale Erkenntnis

Claude Code kann vieles selbst umsetzen – aber es benötigt gute Anweisungen, klare Akzeptanzkriterien, Regeln und Prozesse sowie menschliche Reviews. Genau wie ein neuer Kollege.

Was wir dafür tun

📦

Ein guter Rahmen mit vielen fertigen Hilfsfunktionen und passenden Beispielen. Die KI lernt aus den Templates, wie unser Code aussehen soll.

✅

Test Driven Development (TDD)

Akzeptanzkriterien aus den Tickets werden als Tests formuliert – bevor die Implementierung beginnt. So garantieren wir, dass die KI das liefert, was gefordert ist. Tests sind die härtesten Leitplanken.

🔄

Angepasster Entwicklungsprozess

Früher: Review vor jedem Commit in dev. Heute: Entwicklung ist so schnell, dass menschliche Reviews der Engpass sind. Deshalb: Reviews müssen erst vor dem Merge in den Test-Branch abgeschlossen sein.

Bereich 3

Was wir zukünftig nutzen wollen

Die Entwicklung rast. Diese drei Bereiche stehen als nächstes auf unserer Roadmap.

🤖

KI-Ticket-Zuweisung

Einfache, klar definierte Tickets sollen direkt an die KI gehen – ohne menschlichen Zwischenschritt. Eine zweite, unabhängige KI prüft die Umsetzung automatisch, bevor ein Mensch den finalen Review macht.

Im Entwicklungszyklus bereits als gestrichelter Schritt angedeutet.

📦

Pakete in Templates integrieren

KI-Agenten haben Probleme mit selbstgestrickten, neuen oder wenig verbreiteten Paketen. Wenn der Code direkt im Template liegt statt in einem separaten npm-Paket, hat die KI direkteren Zugriff und versteht ihn besser.

nest-server → nest-server-starter, nuxt-extensions → nuxt-base-starter

🧠

Memory & Langzeitgedächtnis

Wir testen Memory-Plugins und Claude Codes eigenes Auto-Memory. Die KI soll projektübergreifend lernen: Architektur-Entscheidungen, häufige Patterns, vergangene Fehler – wie ein Teamkollege, der Erfahrung sammelt.

Alles entwickelt sich so schnell, dass auch Claude Code bereits eigenes Memory bietet.

Bereich 4

Was ich euch mitgeben möchte

Neun Dinge, die den Unterschied machen.

1

Rasende Geschwindigkeit

Der Eindruck von gestern kann sich heute wandeln. Was vor Monaten war, ist Steinzeit. Dranbleiben ist Pflicht, nicht Kür.

2

Gutes Briefing: Vernünftige Tickets

Klare Beschreibungen, präzise Anforderungen, Kontext. Was für neue Kolleg:innen gilt, gilt auch für die KI.

3

Harte Leitplanken: Akzeptanzkriterien → TDD

Tests sind die härtesten Leitplanken. Sie garantieren, dass geliefert wird, was gefordert ist – unabhängig davon, wer implementiert.

4

Zusätzliches Wissen: Plugin & CLAUDE.md

Regeln, Patterns, Best Practices – alles, was nicht im öffentlichen Trainingsset steht, muss aktiv bereitgestellt werden.

5

Abstand nehmen: /clear

Genau wie wir manchmal einen frischen Blick brauchen, hilft es der KI, mit einem sauberen Kontext neu zu starten.

6

Wiederholte Reviews

Auch uns fallen bei mehrfachem Draufschauen unterschiedliche Dinge auf. Mehrere Review-Runden sind kein Zeichen von Schwäche.

7

Human in the Loop

Entscheidungen und finale Reviews bleiben beim Menschen. Die KI ist der starke Assistent, nicht der Entscheider.

8

Prozesse anpassen

Neue Möglichkeiten erfordern neue Prozesse. Wer mit KI-Speed entwickelt, braucht angepasste Review- und Deployment-Workflows.

9

Erkenntnisse teilen

Erfahrungen austauschen und sammeln – sie fließen in Prozesse und Plugin zurück. Gemeinsam werden wir besser.

Das ist alles nichts Neues. Denn alles, was für eine effiziente KI-Unterstützung gilt, gilt eigentlich schon immer für gute Software­entwicklung.
Gute Tickets, klare Kriterien, saubere Prozesse, Reviews, Wissenstransfer – das waren schon immer die Grundlagen.
